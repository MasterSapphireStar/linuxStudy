# Linux 进程间通信（IPC）完整指南

## 概述

进程间通信（Inter-Process Communication，IPC）是指在不同进程之间传播或交换信息的机制。由于每个进程都有自己独立的地址空间，一个进程不能直接访问另一个进程的内存，因此需要通过操作系统提供的 IPC 机制进行通信。

## Linux 中的 7 种主要 IPC 方式

### 1. 管道（Pipe） - 最简单

### 2. 命名管道（FIFO） - 可用于无关进程

### 3. 消息队列（Message Queue） - 按类型处理

### 4. 共享内存（Shared Memory） - 最快

### 5. 信号量（Semaphore） - 同步控制

### 6. 信号（Signal） - 异步通知

### 7. 套接字（Socket） - 网络通信

---

## 1. 管道（Pipe）

### 基本概念

- **半双工通信**：数据只能单向流动
- **匿名管道**：只能用于有亲缘关系的进程（父子、兄弟）
- **内存实现**：存在于内核内存中，不属于文件系统
- **读后即删**：数据被读取后就从管道中删除

### 工作原理

```
父进程                    管道                    子进程
write(fd[1]) ---------> [Buffer] ---------> read(fd[0])
  写端                   内核缓冲区              读端
```

### 关键函数

```c
int pipe(int fd[2]);
// fd[0]: 读端
// fd[1]: 写端
// 返回: 成功返回0，失败返回-1
```

### 典型使用流程

1. 父进程调用 `pipe()` 创建管道
2. 父进程调用 `fork()` 创建子进程
3. 父进程关闭读端 `fd[0]`，子进程关闭写端 `fd[1]`
4. 父进程写数据到 `fd[1]`，子进程从 `fd[0]` 读数据

### 完整示例

见 `IPC示例代码/01_pipe管道.c`

### 优缺点

**优点**：

- 简单易用，无需复杂配置
- 内核自动管理生命周期
- 适合父子进程简单通信

**缺点**：

- 只能单向通信（双向需要两个管道）
- 只能用于有血缘关系的进程
- 缓冲区大小有限（一般 4KB-64KB）

### 应用场景

- Shell 命令管道：`ls | grep txt`
- 父子进程间简单数据传递
- 过滤器程序设计模式

## 2. 命名管道（FIFO）

### 基本概念

- **有名管道**：以文件形式存在于文件系统中
- **可见性**：任何进程都可以通过文件路径访问
- **持久性**：需要手动删除，否则一直存在
- **仍是半双工**：数据单向流动

### 工作原理

```
进程A                    FIFO文件                  进程B
write() ---------> /tmp/my_fifo ---------> read()
                   (存在于文件系统)
```

### 关键函数

```c
// 创建FIFO
int mkfifo(const char *pathname, mode_t mode);

// 之后使用标准文件操作
int fd = open(pathname, O_RDONLY | O_WRONLY);
write(fd, buf, size);
read(fd, buf, size);
close(fd);
```

### 典型使用流程

1. 一个进程调用 `mkfifo()` 创建 FIFO 文件
2. 读进程以 `O_RDONLY` 方式打开（会阻塞直到有写进程）
3. 写进程以 `O_WRONLY` 方式打开
4. 进行读写操作
5. 关闭文件描述符
6. 删除 FIFO 文件 `unlink()`

### 完整示例

- 写进程：`IPC示例代码/02_fifo命名管道.c`
- 读进程：`IPC示例代码/03_fifo读进程.c`

### 优缺点

**优点**：

- 可用于无亲缘关系的进程
- 以文件形式存在，便于管理和调试
- 支持多个读者和写者

**缺点**：

- 仍然是半双工通信
- 需要手动创建和删除
- 打开 FIFO 时可能阻塞

### 应用场景

- 客户端-服务器通信（单机）
- 日志收集系统
- 进程间命令传递

## 3. 消息队列（Message Queue）

### 基本概念

- **消息链表**：存放在内核中的消息链表
- **有结构**：每个消息有类型和数据
- **按类型读取**：可以选择性地接收特定类型的消息
- **持久性**：消息队列在内核中，进程结束后仍存在

### 工作原理

```
进程A                    内核消息队列                 进程B
msgsnd() ------> [类型1|数据1] ------> msgrcv(类型1)
                 [类型2|数据2] ------> msgrcv(类型2)
                 [类型1|数据3]
```

### 关键函数

```c
// 创建或获取消息队列
int msgget(key_t key, int msgflg);

// 发送消息
int msgsnd(int msqid, const void *msgp, size_t msgsz, int msgflg);

// 接收消息
ssize_t msgrcv(int msqid, void *msgp, size_t msgsz, long msgtyp, int msgflg);

// 控制消息队列
int msgctl(int msqid, int cmd, struct msqid_ds *buf);
```

### 消息结构定义

```c
struct msgbuf {
    long mtype;       // 消息类型，必须>0
    char mtext[128];  // 消息数据
};
```

### msgrcv 的 msgtyp 参数

- **msgtyp = 0**：接收队列中的第一个消息
- **msgtyp > 0**：接收类型等于 msgtyp 的第一个消息
- **msgtyp < 0**：接收类型小于等于|msgtyp|的最小类型消息

### 完整示例

见 `IPC示例代码/04_message_queue消息队列.c`

### 优缺点

**优点**：

- 可以按消息类型读取，实现优先级处理
- 消息有结构，可以携带更多信息
- 不需要考虑同步问题（内核管理）
- 支持异步通信

**缺点**：

- 有消息大小限制（系统相关，一般几 KB）
- 需要手动删除，否则会一直占用资源
- 不适合大数据量传输

### 应用场景

- 任务队列系统
- 优先级消息处理
- 事件驱动系统
- 进程间异步通知

### 实用命令

```bash
# 查看消息队列
ipcs -q

# 删除消息队列
ipcrm -q <msqid>
```

## 4. 共享内存（Shared Memory）

### 基本概念

- **最快的 IPC**：直接访问内存，无需数据复制
- **共享物理内存**：多个进程映射到同一块物理内存
- **需要同步**：必须配合信号量等同步机制
- **高效传输**：适合大量数据传输

### 工作原理

```
进程A地址空间          物理内存          进程B地址空间
0x1000 ---------> [共享内存区] <--------- 0x2000
    |               |                    |
    +--- shmat() ---+--- shmat() --------+
                直接读写，无需复制
```

### 关键函数

```c
// 创建或获取共享内存
int shmget(key_t key, size_t size, int shmflg);

// 映射到进程地址空间
void *shmat(int shmid, const void *shmaddr, int shmflg);

// 分离共享内存
int shmdt(const void *shmaddr);

// 控制共享内存
int shmctl(int shmid, int cmd, struct shmid_ds *buf);
```

### 典型使用流程

1. 调用 `shmget()` 创建共享内存段
2. 调用 `shmat()` 将共享内存映射到进程地址空间
3. 直接通过指针读写共享内存
4. 使用信号量保证同步
5. 调用 `shmdt()` 分离共享内存
6. 调用 `shmctl()` 删除共享内存

### 完整示例

见 `IPC示例代码/05_shared_memory共享内存.c`

### 优缺点

**优点**：

- **速度最快**：直接内存访问，零拷贝
- 适合大量数据传输
- 多个进程可以同时访问

**缺点**：

- **必须配合同步机制**（信号量、互斥锁）
- 需要手动管理内存
- 编程复杂度较高
- 可能出现数据竞争

### 应用场景

- 高性能计算
- 数据库系统（如 Redis、MySQL 缓存）
- 多媒体处理（视频、图像数据）
- 实时系统数据共享

### 实用命令

```bash
# 查看共享内存
ipcs -m

# 删除共享内存
ipcrm -m <shmid>
```

### 重要提示

⚠️ **共享内存必须配合同步机制使用，否则会出现数据竞争！**

## 5. 信号量（Semaphore）

### 基本概念

- **同步工具**：用于进程/线程间的同步和互斥
- **计数器**：本质上是一个整数计数器
- **原子操作**：P/V 操作是原子的，不可被中断
- **配合使用**：通常与共享内存配合使用

### P/V 操作原理

- **P 操作（wait/down）**：信号量减 1，若结果<0 则阻塞
- **V 操作（signal/up）**：信号量加 1，若结果 ≤0 则唤醒一个等待进程

```
P操作：               V操作：
sem--;               sem++;
if (sem < 0)         if (sem <= 0)
    阻塞;                唤醒一个进程;
```

### 关键函数

```c
// 创建或获取信号量
int semget(key_t key, int nsems, int semflg);

// P/V操作
int semop(int semid, struct sembuf *sops, size_t nsops);

// 控制信号量
int semctl(int semid, int semnum, int cmd, ...);
```

### sembuf 结构

```c
struct sembuf {
    unsigned short sem_num;  // 信号量编号
    short sem_op;            // -1:P操作, +1:V操作
    short sem_flg;           // 操作标志
};
```

### 典型使用模式

#### 互斥信号量（初始值=1）

```c
// 初始化为1
semctl(semid, 0, SETVAL, 1);

// 进入临界区
P(semid);  // 获取锁
// ... 临界区代码 ...
V(semid);  // 释放锁
```

#### 同步信号量（初始值=0）

```c
// 进程A
// ... 完成某项工作 ...
V(semid);  // 通知进程B

// 进程B
P(semid);  // 等待进程A完成
// ... 继续工作 ...
```

### 完整示例

见 `IPC示例代码/06_semaphore信号量.c`

### 经典同步问题

#### 1. 生产者-消费者问题

```c
信号量：
- mutex = 1      // 互斥访问缓冲区
- empty = N      // 空位数量
- full = 0       // 产品数量

生产者：
P(empty);        // 等待空位
P(mutex);        // 加锁
// 生产产品
V(mutex);        // 解锁
V(full);         // 产品数+1

消费者：
P(full);         // 等待产品
P(mutex);        // 加锁
// 消费产品
V(mutex);        // 解锁
V(empty);        // 空位数+1
```

#### 2. 读者-写者问题

- 多个读者可以同时读
- 写者必须独占访问
- 写者优先或读者优先

#### 3. 哲学家就餐问题

- 防止死锁
- 公平性保证

### 优缺点

**优点**：

- 提供可靠的同步机制
- 支持原子操作
- 可以解决复杂的同步问题

**缺点**：

- 不传输数据，只做控制
- 使用不当会导致死锁
- 调试困难

### 应用场景

- 共享内存的同步保护
- 资源池管理（数据库连接池）
- 生产者-消费者模型
- 并发控制

### 实用命令

```bash
# 查看信号量
ipcs -s

# 删除信号量
ipcrm -s <semid>
```

## 6. 信号（Signal）

### 基本概念

- **软件中断**：异步通知机制
- **事件通知**：通知进程某个事件发生
- **信息量少**：只携带信号编号
- **不可靠**：信号可能丢失（不排队）

### 常用信号表

| 信号    | 值  | 默认动作  | 说明                 |
| ------- | --- | --------- | -------------------- |
| SIGHUP  | 1   | 终止      | 挂起（终端断开）     |
| SIGINT  | 2   | 终止      | 中断（Ctrl+C）       |
| SIGQUIT | 3   | 终止+core | 退出（Ctrl+\）       |
| SIGKILL | 9   | 终止      | 强制终止（不可捕获） |
| SIGSEGV | 11  | 终止+core | 段错误               |
| SIGTERM | 15  | 终止      | 终止信号（可捕获）   |
| SIGCHLD | 17  | 忽略      | 子进程状态改变       |
| SIGCONT | 18  | 继续      | 继续执行             |
| SIGSTOP | 19  | 停止      | 停止（不可捕获）     |
| SIGUSR1 | 10  | 终止      | 用户自定义 1         |
| SIGUSR2 | 12  | 终止      | 用户自定义 2         |

### 关键函数

#### 注册信号处理函数

```c
// 简单方式（不推荐）
void (*signal(int sig, void (*func)(int)))(int);

// 推荐方式
int sigaction(int signum, const struct sigaction *act,
              struct sigaction *oldact);
```

#### 发送信号

```c
int kill(pid_t pid, int sig);      // 向指定进程发送
int raise(int sig);                 // 向自己发送
unsigned int alarm(unsigned int s); // 定时发送SIGALRM
void abort(void);                   // 向自己发送SIGABRT
```

#### 信号集操作

```c
int sigemptyset(sigset_t *set);     // 清空信号集
int sigfillset(sigset_t *set);      // 填满信号集
int sigaddset(sigset_t *set, int signum);   // 添加信号
int sigdelset(sigset_t *set, int signum);   // 删除信号
int sigismember(const sigset_t *set, int signum); // 测试
```

#### 信号屏蔽

```c
int sigprocmask(int how, const sigset_t *set, sigset_t *oldset);
// how: SIG_BLOCK, SIG_UNBLOCK, SIG_SETMASK
```

### 信号处理流程

```
1. 信号产生（kill、硬件异常、终端按键等）
   ↓
2. 信号未决（暂时存储）
   ↓
3. 信号递达（检查是否阻塞）
   ↓
4. 执行信号处理函数或默认动作
```

### 完整示例

见 `IPC示例代码/07_signal信号.c`

### 信号处理函数注意事项

#### 可重入函数

信号处理函数必须是可重入的，避免调用：

- `printf`、`sprintf` 等（使用 `write` 代替）
- `malloc`、`free`
- 其他不可重入函数

#### 安全函数列表

```c
write()、read()、_exit()、
signal()、sigaction()、
wait()、waitpid()、
pause()、sleep()
```

### 优缺点

**优点**：

- 异步通信，不阻塞进程
- 简单高效
- 系统原生支持

**缺点**：

- 携带信息量少（只有信号类型）
- 信号可能丢失（不排队）
- 某些信号不可捕获（SIGKILL、SIGSTOP）
- 处理函数限制多

### 应用场景

- 进程控制（启动、停止、重启）
- 异常处理（段错误、非法指令）
- 定时器（SIGALRM）
- 用户中断处理（Ctrl+C）
- 守护进程管理

### 实用命令

```bash
# 向进程发送信号
kill -SIGUSR1 <pid>
kill -9 <pid>        # 强制终止

# 查看信号
kill -l              # 列出所有信号
```

## 7. 套接字（Socket）

### 基本概念

- **最通用的 IPC**：既可本地通信，也可网络通信
- **双向通信**：全双工通信
- **多协议支持**：TCP、UDP 等
- **跨主机**：可以在不同机器间通信

### 套接字类型

#### 按通信域分类

1. **AF_UNIX / AF_LOCAL**：本地通信（Unix 域套接字）
2. **AF_INET**：IPv4 网络通信
3. **AF_INET6**：IPv6 网络通信

#### 按类型分类

1. **SOCK_STREAM**：流式套接字（TCP）
   - 可靠、有序、面向连接
   - 适合传输大量数据
2. **SOCK_DGRAM**：数据报套接字（UDP）
   - 不可靠、无连接
   - 适合实时数据传输

### TCP 通信流程

#### 服务器端

```c
1. socket()   - 创建套接字
2. bind()     - 绑定地址和端口
3. listen()   - 监听连接
4. accept()   - 接受连接（阻塞）
5. read/write - 读写数据
6. close()    - 关闭连接
```

#### 客户端

```c
1. socket()   - 创建套接字
2. connect()  - 连接服务器
3. read/write - 读写数据
4. close()    - 关闭连接
```

### 关键函数

#### 创建套接字

```c
int socket(int domain, int type, int protocol);
// domain: AF_UNIX, AF_INET, AF_INET6
// type: SOCK_STREAM, SOCK_DGRAM
// protocol: 通常为0（自动选择）
```

#### 绑定地址

```c
int bind(int sockfd, const struct sockaddr *addr, socklen_t addrlen);
```

#### 监听连接

```c
int listen(int sockfd, int backlog);
// backlog: 等待连接队列的最大长度
```

#### 接受连接

```c
int accept(int sockfd, struct sockaddr *addr, socklen_t *addrlen);
// 返回新的套接字文件描述符
```

#### 连接服务器

```c
int connect(int sockfd, const struct sockaddr *addr, socklen_t addrlen);
```

#### 读写数据

```c
ssize_t read(int fd, void *buf, size_t count);
ssize_t write(int fd, const void *buf, size_t count);

// 或者使用
ssize_t recv(int sockfd, void *buf, size_t len, int flags);
ssize_t send(int sockfd, const void *buf, size_t len, int flags);
```

### Unix 域套接字地址结构

```c
struct sockaddr_un {
    sa_family_t sun_family;  // AF_UNIX
    char sun_path[108];      // 路径名
};
```

### Internet 域套接字地址结构

```c
struct sockaddr_in {
    sa_family_t sin_family;     // AF_INET
    in_port_t sin_port;         // 端口号（网络字节序）
    struct in_addr sin_addr;    // IP地址
};
```

### 完整示例

见 `IPC示例代码/08_socket套接字.c`（Unix 域套接字）

### TCP 网络编程示例

#### 服务器

```c
int server_fd = socket(AF_INET, SOCK_STREAM, 0);

struct sockaddr_in addr;
addr.sin_family = AF_INET;
addr.sin_port = htons(8080);  // 网络字节序
addr.sin_addr.s_addr = INADDR_ANY;

bind(server_fd, (struct sockaddr *)&addr, sizeof(addr));
listen(server_fd, 5);

int client_fd = accept(server_fd, NULL, NULL);
// 读写数据...
close(client_fd);
close(server_fd);
```

#### 客户端

```c
int sock_fd = socket(AF_INET, SOCK_STREAM, 0);

struct sockaddr_in addr;
addr.sin_family = AF_INET;
addr.sin_port = htons(8080);
inet_pton(AF_INET, "127.0.0.1", &addr.sin_addr);

connect(sock_fd, (struct sockaddr *)&addr, sizeof(addr));
// 读写数据...
close(sock_fd);
```

### 常见问题

#### 1. 粘包问题

TCP 是流式协议，可能出现粘包，解决方案：

- 固定长度消息
- 消息头+长度
- 特殊分隔符

#### 2. 字节序问题

网络字节序是大端，需要转换：

```c
htons()  // host to network short
htonl()  // host to network long
ntohs()  // network to host short
ntohl()  // network to host long
```

#### 3. TIME_WAIT 问题

设置地址复用：

```c
int opt = 1;
setsockopt(fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));
```

### 优缺点

**优点**：

- 功能最强大，支持网络通信
- 双向通信
- 支持多种协议
- 跨平台、跨主机

**缺点**：

- 编程复杂度高
- 本地通信性能不如共享内存
- 需要处理网络问题（超时、断线等）

### 应用场景

- Web 服务器（HTTP 服务器）
- 数据库客户端-服务器
- 分布式系统
- 微服务架构
- 聊天室、游戏服务器
- RPC 框架

## 8. IPC 方式全面对比

### 选择决策树

```
需要网络通信？
├─ 是 → Socket
└─ 否 → 本地IPC
    │
    ├─ 只是进程控制/通知？
    │  └─ 是 → Signal
    │
    ├─ 只需要同步控制？
    │  └─ 是 → Semaphore
    │
    ├─ 数据量大（>1MB）？
    │  ├─ 是 → Shared Memory + Semaphore
    │  └─ 否 → 继续
    │
    ├─ 需要消息类型/优先级？
    │  └─ 是 → Message Queue
    │
    ├─ 有父子关系？
    │  ├─ 是 → Pipe
    │  └─ 否 → Named Pipe
    │
    └─ 其他 → 根据具体需求选择
```

### 使用建议

#### 场景 1：父子进程简单通信

**推荐**：管道（Pipe）

```c
适用：ls | grep txt
特点：简单、直接、无需清理
```

#### 场景 2：无关进程少量数据

**推荐**：命名管道（FIFO）或消息队列

```c
FIFO：简单通信
消息队列：需要消息类型或优先级
```

#### 场景 3：大量数据高性能传输

**推荐**：共享内存 + 信号量

```c
共享内存：零拷贝，速度最快
信号量：保证数据一致性
```

#### 场景 4：进程控制和通知

**推荐**：信号（Signal）

```c
适用：启动、停止、重启、异常处理
特点：异步、简单、系统支持
```

#### 场景 5：网络通信或分布式

**推荐**：套接字（Socket）

```c
支持TCP/UDP
跨主机通信
灵活性最高
```

## 9. 实战技巧

### 1. 共享内存+信号量组合

```c
// 最常用的高性能IPC组合
// 1. 创建共享内存
shmid = shmget(key, size, IPC_CREAT | 0666);
ptr = shmat(shmid, NULL, 0);

// 2. 创建信号量（初始值1，用于互斥）
semid = semget(key, 1, IPC_CREAT | 0666);
semctl(semid, 0, SETVAL, 1);

// 3. 使用
P(semid);  // 加锁
// 访问共享内存
V(semid);  // 解锁
```

### 2. 管道+信号组合

```c
// 用于进程控制和数据传输
// 管道传输数据，信号通知事件
pipe(fd);
signal(SIGUSR1, handler);

// 子进程
write(fd[1], data, size);
kill(getppid(), SIGUSR1);  // 通知父进程

// 父进程
void handler(int sig) {
    read(fd[0], buf, size);  // 读取数据
}
```

### 3. 消息队列优先级处理

```c
// 使用消息类型实现优先级
struct msg {
    long type;  // 1=高优先级, 2=中, 3=低
    char text[128];
};

// 优先处理高优先级消息
msgrcv(msgid, &msg, size, 1, IPC_NOWAIT);  // 先尝试type=1
if (errno == ENOMSG) {
    msgrcv(msgid, &msg, size, 2, IPC_NOWAIT);  // 再尝试type=2
}
```

### 4. Socket 错误处理

```c
// 超时设置
struct timeval tv;
tv.tv_sec = 5;
tv.tv_usec = 0;
setsockopt(fd, SOL_SOCKET, SO_RCVTIMEO, &tv, sizeof(tv));

// 非阻塞模式
int flags = fcntl(fd, F_GETFL, 0);
fcntl(fd, F_SETFL, flags | O_NONBLOCK);

// 地址复用
int opt = 1;
setsockopt(fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));
```

### 5. IPC 资源清理

```c
// 创建时注册清理函数
void cleanup() {
    // 删除消息队列
    msgctl(msgid, IPC_RMID, NULL);

    // 删除共享内存
    shmctl(shmid, IPC_RMID, NULL);

    // 删除信号量
    semctl(semid, 0, IPC_RMID);

    // 删除FIFO
    unlink("/tmp/my_fifo");
}

atexit(cleanup);  // 程序退出时自动清理
signal(SIGINT, signal_cleanup);  // Ctrl+C时清理
```

## 10. 常见错误和调试

### 错误 1：忘记关闭文件描述符

```c
// ❌ 错误
pipe(fd);
fork();
// 父子进程都有fd[0]和fd[1]

// ✅ 正确
pipe(fd);
if (fork() > 0) {
    close(fd[0]);  // 父进程关闭读端
} else {
    close(fd[1]);  // 子进程关闭写端
}
```

### 错误 2：共享内存没有同步

```c
// ❌ 错误：数据竞争
ptr = shmat(shmid, NULL, 0);
sprintf(ptr, "data");  // 多个进程同时写

// ✅ 正确：加锁保护
P(semid);
sprintf(ptr, "data");
V(semid);
```

### 错误 3：信号处理函数不安全

```c
// ❌ 错误：printf不可重入
void handler(int sig) {
    printf("Signal received\n");  // 危险！
}

// ✅ 正确：使用write
void handler(int sig) {
    write(STDOUT_FILENO, "Signal received\n", 16);
}
```

### 错误 4：忘记清理 IPC 资源

```c
// ❌ 错误：资源泄露
msgget(key, IPC_CREAT | 0666);
// 程序异常退出，消息队列仍然存在

// ✅ 正确：注册清理函数
msgid = msgget(key, IPC_CREAT | 0666);
atexit(cleanup);
signal(SIGINT, cleanup_and_exit);
```

## 11. 性能优化建议

### 1. 选择合适的 IPC 方式

- 大数据 → 共享内存
- 小数据 → 管道/消息队列
- 网络 → Socket
- 控制 → 信号

### 2. 减少数据拷贝

```c
// 避免多次拷贝
// 用户空间 → 内核 → 用户空间

// 使用共享内存直接访问
ptr = shmat(shmid, NULL, 0);
// 直接操作ptr，零拷贝
```

### 3. 使用缓冲

```c
// 批量读写，减少系统调用
char buf[4096];
while ((n = read(fd, buf, sizeof(buf))) > 0) {
    process(buf, n);
}
```

### 4. 异步 I/O

```c
// 使用select/poll/epoll
fd_set readfds;
FD_ZERO(&readfds);
FD_SET(fd, &readfds);
select(fd + 1, &readfds, NULL, NULL, &timeout);
```

### 5. 内存对齐

```c
// 共享内存中的结构体对齐
struct __attribute__((aligned(64))) data {
    int value;
    char text[60];
};
```

## 12. 总结

### IPC 方式速查表

| 需求         | 推荐方案             | 理由     |
| ------------ | -------------------- | -------- |
| 父子进程通信 | Pipe                 | 简单直接 |
| 任意进程通信 | FIFO / Message Queue | 灵活方便 |
| 大数据传输   | Shared Memory        | 速度最快 |
| 进程同步     | Semaphore            | 专门用途 |
| 进程控制     | Signal               | 系统支持 |
| 网络通信     | Socket               | 唯一选择 |
| 高优先级     | Message Queue        | 支持类型 |

### 学习路线建议

1. **入门**：管道、命名管道、信号
2. **进阶**：消息队列、共享内存+信号量
3. **高级**：套接字编程、性能优化

### 最佳实践

1. ✅ 根据场景选择合适的 IPC
2. ✅ 共享内存必须配合同步机制
3. ✅ 及时清理 IPC 资源
4. ✅ 错误处理要完善
