下面直接用中文解释你的疑问——为什么看起来 if 似乎“执行了两次”。

主要原因：fork 会把当前进程复制出一个新的子进程，复制的是“从 fork 返回处继续执行”的那一份程序状态（代码、寄存器、堆栈和用户空间的内存副本）。也就是说，fork 之后程序的执行流在父进程和子进程中各自继续，fork 在父进程中返回子进程的 PID（>0），在子进程中返回 0。因为父子进程都是从 fork 返回点继续执行 if 判断，所以看上去 if “被执行两次”，但实际上是两份进程各自独立执行一次判断，结果不同（父进程进入 else 分支，子进程进入 pid==0 的分支），因此会各打印一次。

更具体的要点：
- fork 后你会有两个进程（父 + 子），它们执行同一段代码但内存是独立的。对你代码的 if 分支来说：
  - 在父进程：pid > 0 -> 执行父进程的 printf（打印父进程信息）。
  - 在子进程：pid == 0 -> 执行子进程的 printf（打印子进程信息）。
- 因此打印两行是正常、预期的行为（父子各一行）。顺序是不确定的（子可能先打印，也可能父先打印），因为它们是并发执行的。

关于输出重复/缓冲的额外说明（常见疑惑）：
- 如果在 fork 之前有未刷新（未换行或未 fflush）的 stdout 缓冲区内容，那么 fork 会把缓冲区的“拷贝”也复制到子进程，导致缓冲区的内容可能在父子进程中都被输出，出现看似重复的同一条文本。这和上面“每个进程各自打印一次不同信息”是两种不同的重复来源。避免方法是 fork 之前调用 fflush(stdout) 或使用行缓冲（在终端通常是自动换行刷新）或将输出直接写到 stderr（stderr 通常是无缓冲的）。
- 你代码里第一行有 printf("fork之前的代码，在父进程中执行，pid=%d\n", getpid()); 因为这一行以换行结尾，通常会被刷新，所以一般不会被复制输出两次。但如果你去掉换行或输出到文件，就要注意 flush 问题。

如何控制打印顺序或让父进程等子进程结束后再打印：
- 在父进程分支使用 wait()/waitpid() 等等待子进程结束，这样可以保证父进程在子进程完成后再继续。例如：
  - 在父分支添加：
    #include <sys/wait.h>
    ...
    else {
        wait(NULL); // 等待子进程结束
        printf("父进程执行的代码，pid=%d, 创建的子进程的pid=%d\n", getpid(), pid);
    }
  这样父进程会先阻塞直到子进程退出，打印顺序更可控。
- 如果你想在父进程不打印只让子进程打印（或反过来），就在对应分支里只保留你想要的打印语句。

示例可能的输出（原始代码不等待）：
子进程执行的代码，pid=12345, 父进程的pid=12344
父进程执行的代码，pid=12344, 创建的子进程的pid=12345

（或顺序相反，取决于调度）

总结（一句话）：if 之所以“看起来执行两次”，是因为 fork 之后程序在父子两个独立进程中各自继续执行，从而导致各自做一次判断并打印各自的消息。这是 fork 的正常行为，不是 bug。
